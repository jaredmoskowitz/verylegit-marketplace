# AGENTS.md

Guidance for AI coding agents (Codex, Cursor, Claude Code, etc.) working in this repo.

## What this is

A [Claude Code plugin marketplace](https://docs.claude.com/en/docs/claude-code/plugins). Each subdirectory under [plugins/](plugins/) is a self-contained plugin. The marketplace manifest lives in [.claude-plugin/marketplace.json](.claude-plugin/marketplace.json).

## Repo layout

```
.claude-plugin/
  marketplace.json          # Lists every plugin in this marketplace
plugins/
  <plugin-name>/
    .claude-plugin/
      plugin.json           # Plugin manifest (required)
    skills/                 # Optional: Claude Code skills (SKILL.md files)
    commands/               # Optional: slash command markdown files
    mcp/                    # Optional: Node.js MCP server
      package.json
      server.mjs
    hooks/                  # Optional: Claude Code hook configs
    hooks-handlers/         # Optional: hook handler scripts
README.md                   # User-facing documentation per plugin
LICENSE                     # MIT
```

## Hard rules

- **One plugin, one job.** Plugins should be narrow. If a plugin starts trying to do two unrelated things, split it.
- **Every plugin needs a `plugin.json`** at `plugins/<name>/.claude-plugin/plugin.json` with at minimum: `name`, `version`, `description`, `author`. If the plugin ships skills, commands, or MCP servers, declare them via `"skills"`, `"commands"`, `"mcp"` keys (see existing plugins for examples).
- **Every new plugin must be added to [.claude-plugin/marketplace.json](.claude-plugin/marketplace.json)** with a `name`, `description`, `source` (`./plugins/<name>`), and `category`.
- **Every new plugin needs a section in [README.md](README.md).** Match the structure of existing plugin sections: short tagline, what it does, usage examples, requirements (if any).
- **Skill triggering is half the product.** Every `SKILL.md` description must clearly state when the skill should activate — vague descriptions cause skills to never fire.

## MCP servers

- Node.js 18+ (ESM, `.mjs` files).
- Each MCP plugin's `mcp/` directory has its own `package.json`. Don't share dependencies across plugins via a root `node_modules`.
- Servers communicate via stdio. Don't add HTTP transports unless the user explicitly asks.
- macOS-specific MCPs (`bear-notes`, `ios-simulator`) document their requirements in README.md and in the plugin's `SKILL.md`.

## Slash commands

- Live in `plugins/<name>/commands/<command>.md`.
- Each command file is self-contained — it should make sense without external context.
- Commands invoked as `/<name>` must match the filename.

## Adding a new plugin (checklist)

1. Create `plugins/<new-plugin>/.claude-plugin/plugin.json` with name/version/description/author.
2. Add the plugin's content (`skills/`, `commands/`, `mcp/`, etc.) referenced from `plugin.json`.
3. Add an entry to `.claude-plugin/marketplace.json`.
4. Add a section to [README.md](README.md) with usage examples.
5. If the plugin has system requirements (macOS, Node, Xcode), spell them out in README and SKILL.md.
6. Test installation: `claude plugins install <name>@verylegit-marketplace`.

## Style

- Voice: dry, occasionally feral. The README and plugin descriptions have personality (see `wtf`, `captain`). Don't sand it off.
- Don't add corporate boilerplate, "AI-generated by..." footers, or filler "Disclaimers" sections.
- Tables in READMEs use `| Column | Column |` pipes, no HTML.

## What not to do

- No telemetry. No analytics. No phone-home.
- No bundled binaries. Plugins must be readable source.
- No plugin dependencies on other plugins in this marketplace — each one stands alone.
